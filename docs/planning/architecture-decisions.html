Architecture Planning · Decision Record

pdf → cbz

An in-browser, offline-first converter that turns zine PDFs into CBZ comic archives — no server, no upload. This document lays the architectural choices side-by-side so each can be approved before any code is written.

Status  Decisions locked Date  2026-06-03 License  AGPL-3.0-or-later Decisions  D1–D8

Decisions locked · 2026-06-03

  • D1 Engine — Hybrid extract-or-render (extract single-image/JPEG pages, render the rest)
  • D2 UI stack — Vanilla TS + Vite
  • D3 ZIP — fflate; adaptive light DEFLATE when affordable, else STORE
  • D4 Concurrency — bounded worker pool sized from measured runtime capability
  • D5 Delivery — Blob + anchor baseline; File System Access streaming when present
  • D6 Rendered format — WebP (+ JPEG fallback); extracted pages keep original bytes
  • D7 pdf.js — bundled first-party (no CDN)
  • D8 Metadata — ComicInfo.xml at zip root; first page = FrontCover; user-editable fields

The ledger above is the decision of record; the badges on the cards below mark each option's recommendation. Behavior adapts to measured runtime capability, with no mobile/desktop hard-coding. Provisional defaults: ~1600px long edge + soft warning · warn-and-continue on bad pages · PWA as a fast-follow.

00 · Fixed Requirements

What's already decided

These are settled inputs from the brief and clarifying Q&A. The options further down are constrained by them.

PDF content

Mixed. Some pages are single full-page scans; others are designed vector/text layouts. The engine must handle both.

Targets

Desktop evergreen + Mobile Chrome (required). Mobile Safari is best-effort and may be dropped for Phase 1.

Workload

One PDF at a time. Simple memory model; a single dedicated worker is sufficient.

Output controls

Smart defaults for v1. Advanced knobs (format, quality, DPI, page range) are a later phase.

Hosting

GitHub Pages — a static site. No backend exists or will. Deployed via a GitHub Actions Pages workflow established in Phase 1, so the build runs in the runner and every change is continuously deployable.

Offline / PWA

Installable, offline-capable. Driven by (a) offline use and (b) minimizing network traffic — large PDFs are never uploaded.

Named assumptions — flag if wrong

  • A1. Zero network requests after the app's own assets load; no telemetry. A service worker + manifest are in-scope goals; the SW caches the app shell & deps but never caches or transmits user files.
  • A2. PDF parsing/rendering uses pdf.js (Mozilla), bundled, not loaded from a CDN.
  • A3. A CBZ is a ZIP of page images in zero-padded order (0001.jpg, 0002.jpg, …); no extra metadata required by readers.
  • A4. Build tooling is Vite; language is TypeScript (strict). The UI framework remains open — see D2.
01 · Decisions

Eight choices, side-by-side

D1

Conversion engine

Your pick: —

The headline tradeoff: ship-fast universality now vs. best fidelity for scans. Recommendation is to phase it.

✓ your pick Recommended · v1

Render-all

Rasterize every page via pdf.js to an image, regardless of its contents.

    Pros
  • +Universal — one code path, works on any PDF.
  • +Simplest pipeline; fastest to ship.
    Cons
  • −Re-encodes embedded scans → quality loss + larger files.
  • −Must pick a rasterization DPI for everything.

Risk  Peak memory on large / high-DPI pages, esp. mobile.

✓ your pick Target · v2

Hybrid extract-or-render

Detect single full-page-image pages → extract the original bytes losslessly; render everything else.

    Pros
  • +Best fidelity for scans; smaller output.
  • +Faster on image-only pages (no re-encode).
    Cons
  • −Detection edge cases; two code paths.
  • −Must handle exotic codecs (JBIG2 / JPEG2000).

Risk  Misdetection produces a wrong-looking page.

✓ your pick Rejected

Extract-only

Pull embedded image bytes only; never render.

    Pros
  • +Lossless and fast where it applies.
    Cons
  • −Fails entirely on vector / text pages.

Why out  Incompatible with the mixed-content requirement.

D3

ZIP library

Your pick: —

CBZ images are already compressed, so the archive uses STORE (no recompression). The differentiator is size, speed & memory.

✓ your pick Recommended

fflate

~8 KB, fastest pure-JS zipper, streaming API.

    Pros
  • +Tiny, fast, tree-shakeable, worker-friendly.
  • +Streaming store → low peak memory.
    Cons
  • −In-memory by default (mitigated via its stream API).

Risk  Low.

✓ your pick Option

zip.js

Streaming zipper that can write straight to disk.

    Pros
  • +Streams to disk (File System Access) — lowest memory.
    Cons
  • −Larger & more complex; FSA is desktop-only anyway.

Risk  Complexity for a gain mobile can't use.

✓ your pick Option

JSZip

The familiar, widely-used zip library.

    Pros
  • +Popular, well-documented, easy.
    Cons
  • −Heavier & slower; higher memory than fflate.

Risk  Memory pressure on mobile.

D4

Worker architecture

Your pick: —

Throughput vs. peak memory. One-PDF-at-a-time plus a mobile target pushes hard toward simplicity.

✓ your pick Recommended

Single dedicated worker

One worker runs a sequential page pipeline.

    Pros
  • +Matches one-PDF-at-a-time; simple memory model.
  • +Easy progress & cancellation.
    Cons
  • −No parallel render — underuses desktop cores.

Risk  Low. Bounded parallelism can be added later, desktop-gated.

✓ your pick Option

Worker pool (parallel)

Render multiple pages concurrently across workers.

    Pros
  • +Faster on multi-core desktop.
    Cons
  • −N× peak memory; more coordination.

Risk  Out-of-memory on mobile.

D5

Output delivery

Your pick: —

Universality vs. peak memory. The two aren't exclusive — baseline everywhere, enhance where supported.

✓ your pick Recommended · baseline

Blob + anchor download

Assemble the ZIP, hand a Blob to a download link.

    Pros
  • +Universal, including Mobile Chrome.
  • +Dead simple.
    Cons
  • −Whole ZIP held in memory before save.

Risk  Memory ceiling for very large CBZ (ties to D3).

✓ your pick Enhancement · later

File System Access streaming

Stream zip entries straight to a chosen file on disk.

    Pros
  • +Lowest peak memory; no giant in-memory Blob.
    Cons
  • −Chromium desktop only — absent on Firefox/Safari/mobile.

Risk  Not a baseline — must sit behind feature detection.

D6

Rendered image format

Your pick: —

Applies to rasterized pages only; extracted pages keep their original bytes. Size vs. reader compatibility.

✓ your pick Recommended

WebP (+ JPEG fallback)

Encode pages to WebP; fall back to JPEG where unsupported.

    Pros
  • +~25–35% smaller than JPEG at similar quality.
  • +Encodable via canvas in all target browsers.
    Cons
  • −Some older CBZ readers don't decode WebP.

Risk  Reader compatibility — see Q O4.

✓ your pick Option

JPEG

Encode every rendered page to JPEG.

    Pros
  • +Universal reader support; small.
    Cons
  • −Larger than WebP; artifacts on line art / text.

Risk  Low; safest for compatibility.

✓ your pick Option

PNG

Lossless encode of every rendered page.

    Pros
  • +Lossless; ideal for crisp line art / text.
    Cons
  • −Much larger for photographic scans.

Risk  Archive bloat on photo-heavy zines.

D7

pdf.js delivery

Your pick: —

Effectively forced by the offline/no-network constraint — shown for completeness.

✓ your pick Required

Bundled with the app

Ship pdf.js (and its worker) as first-party assets.

    Pros
  • +Works offline; no third-party requests; PWA-cacheable.
  • +Matches the no-network / privacy ethos.
    Cons
  • −Larger initial payload (lazy-load + SW cache to mitigate).

Risk  Bundle size — manage via code-splitting.

✓ your pick Rejected

Loaded from a CDN

Fetch pdf.js from a third-party host at runtime.

    Pros
  • +Smaller repo; potential shared cache.
    Cons
  • −Breaks offline use; adds a network request & third party.

Why out  Directly violates the offline / no-network goals.

D8

Metadata & cover

Your pick: —

Readers (Tachiyomi/Komga/Kavita) read ComicInfo.xml; the first page is tagged as the cover, and a form fills what the PDF can't.

✓ your pick Decided

ComicInfo.xml + FrontCover

Write ComicInfo.xml at the zip root; mark page 0 as FrontCover; pre-fill from the PDF and let the user override.

    Pros
  • +Series/title/credits show in readers; first page becomes the cover.
  • +Lightweight, widely-supported standard.
    Cons
  • −Needs a small metadata form + field mapping.

Risk  Low. Omit unknown fields rather than emit empties.

✓ your pick Rejected

Images only

Ship a bare ZIP of images, no metadata file.

    Pros
  • +Simplest possible output.
    Cons
  • −No series/title info; reader guesses the cover.

Why out  Loses metadata the user explicitly wants.

02 · Risks & Edge Cases

What could break later

Flagged now so they shape the design rather than ambush it. The memory risk is primary given a render-all v1 on mobile.

Large PDFs vs. mobile memory Primary

High page counts / high DPI can exhaust a phone's memory under render-all v1. Mitigate: stream each page into the archive and release buffers immediately.

Unsupported image codecs

pdf.js decode support for JBIG2 / JPEG2000 / CCITT (common in scans) is uneven; may force a render fallback or fail a page.

Encrypted / password PDFs

Need a clear prompt or a clean failure, not a silent crash.

Corrupt / malformed PDFs

Must degrade gracefully and report which page/file failed.

Rasterization DPI choice

Too low = blurry pages; too high = out-of-memory. Needs a sensible default and possibly an adaptive cap (see Q O3).

Colorspace & transparency

CMYK and transparent regions can shift colors or composite wrong if flattened naively against the wrong background.

OffscreenCanvas / Safari quirks

Feature-detect and provide a main-thread fallback. Mobile Safari being best-effort means quirks can be a known limitation, not a blocker.

PWA cache correctness

Service worker must cache the app shell & pdf.js for true offline use — and never cache or transmit a user's PDF.

Page order & naming

Zero-padded filenames must preserve reading order; some zines read RTL.

Filename sanitization

Derive a safe .cbz name from the source across OSes.

Cancel / interrupt

Cancelling mid-convert must free buffers and leave no partial download.

Full-ZIP memory ceiling

Holding the finished archive as a Blob before saving caps max output size on the baseline path (ties to D3 & D5).
